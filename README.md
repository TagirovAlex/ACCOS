## 📋 1. Общие Положения

### 1.1. Назначение Системы
ACCOS — это бэкенд-сервис, который выступает в роли оркестратора между пользователем (через фронтенд), LLM (интеграция с LMStudio) и ComfyUI. Он управляет контекстом чатов, выполняет вызовы генерации по заданным шаблонам и строго контролирует расход внутренней валюты пользователя.

### 1.2. Архитектурный Стек
*   **Бэкенд:** Python (FastAPI) — **Обязательно использование асинхронности (`async`/`await`)**.
*   **База Данных:** PostgreSQL — Для обеспечения транзакционной целостности данных.
*   **Фронтенд:** React / Vue.js (Рекомендуется React).
*   **AI Интеграция:** **LMStudio API Client** (для LLM) и **Прямой HTTP Клиент** (для ComfyUI API). *LangChain/Transformers используются как вспомогательные инструменты для управления цепочками, но основная логика оркестрации должна быть в FastAPI.*
*   **Авторизация:** LDAP/AD через `python-ldap`.

### 1.3. Требования к Доступу и Безопасности
1.  **Авторизация:** Все запросы должны проходить аутентификацию (JWT токен после успешного входа).
2.  **Авторизация Групп:** Система должна проверять принадлежность пользователя к определенным группам AD.
3.  **Начальный Баланс:** При первом логине, баланс устанавливается согласно правилам группы (см. п. 4.1).

## ⚙️ 2. Функциональные Требования (User Stories)

### 2.1. Модуль Чат-Ассистента (LLM Interaction - LMStudio)
*   **FR-CHAT-001:** Пользователь должен иметь возможность начать новый чат или продолжить существующий.
*   **FR-CHAT-002:** Должна быть реализована поддержка **Системного Промпта**, который применяется ко всем сообщениям в рамках одного чата.
*   **FR-CHAT-003:** Система должна поддерживать историю диалога (память) для каждого сеанса.
*   **FR-CHAT-004:** При отправке сообщения, система должна: а) Списать токены; б) Вызвать LMStudio API; в) Вернуть ответ пользователю.

### 2.2. Модуль Генерации Изображений (ComfyUI Integration)
*   **FR-IMG-001:** Пользователь должен иметь возможность выбрать **предустановленный Workflow** из списка.
*   **FR-IMG-002:** Должна быть возможность передать текстовый промпт для генерации по выбранному workflow.
*   **FR-IMG-003 (Z-Image):** Пользователь должен иметь возможность загрузить референсное изображение и дать описание редактирования, используя базовый workflow. **(Важно: Z-Image генерирует Изображение на основе промпта)**.
*   **FR-IMG-004 (Qwen Image Edit - Batch/Multi):** Пользователь должен иметь возможность передать список из N изображений и единый промпт для их пакетного редактирования через модель Qwen Image Edit. **(Важно: Это специализированный редактор, который принимает 1, 2 или 3 референса)**.

### 2.3. Модуль Генерации Видео (ComfyUI Integration)
*   **FR-VID-001:** Пользователь должен иметь возможность выбрать workflow генерации видео (Ltx 2.3).
*   **FR-VID-002:** Должна быть возможность передать промпт и задать параметры (например, желаемую длительность/разрешение).

### 2.4. Модуль Оркестрации (Workflow Chaining)
*   **FR-ORCH-001:** После успешной генерации изображения (из FR-IMG), пользователь должен иметь возможность **передать его в редактирование** (вызвать Z-Image с этим изображением как референсом).
*   **FR-ORCH-002:** Пользователь должен иметь возможность **передать изображение в видеогенерацию**, используя его как начальный кадр/референс.

## 💰 3. Экономический Модуль (Token & Currency Management)

### 3.1. Валюта
*   **Единица:** Внутренняя валюта ("Кредиты").
*   **Хранение:** Поле `balance` в таблице `users`.

### 3.2. Расчет Стоимости LLM (Чат)
Стоимость рассчитывается на основе токенов:
$$\text{Cost}_{\text{LLM}} = (\text{InputTokens} \times \text{Rate}_{\text{In}}) + (\text{OutputTokens} \times \text{Rate}_{\text{Out}})$$

### 3.3. Расчет Стоимости Генерации Изображений (ComfyUI)
Стоимость зависит от типа операции:
*   **Z-Image (Генерация):** $\text{Cost}_{\text{Gen}} = \text{BaseCost} + (\text{Width} \times \text{Height}) \times \text{Rate}_{\text{Pixel}}$
*   **Qwen Image Edit:** $\text{Cost}_{\text{Edit}} = \text{BaseCost}_{\text{Edit}} + (\text{Avg\_Ref\_Size} \times \text{Height}) \times \text{Rate}_{\text{Pixel}}$ (где Avg\_Ref\_Size — средний размер референса).

### 3.4. Расчет Стоимости Генерации Видео (ComfyUI)
$$\text{Cost}_{\text{Video}} = \text{BaseCost}_{\text{Vid}} + (\text{Resolution} \times \text{Duration}) \times \text{Rate}_{\text{Sec}}$$

### 3.5. Транзакционный Процесс (Обязательно)
1.  **Транзакция:** Все операции должны быть атомарными в PostgreSQL.
2.  **Проверка/Списание:** Перед вызовом API, проверить баланс и списать стоимость.
3.  **Выполнение:** Вызвать LLM/ComfyUI.
4.  **Фиксация:** **COMMIT** транзакции при успехе, **ROLLBACK** при ошибке.

## 🛠️ 4. Административные Функции (Admin Panel)

Администратор должен иметь доступ к специальному API эндпоинту (`/admin/*`) для управления системой:

### 4.1. Управление Пользователями и Материалами
*   **Просмотр Всех Чатов:** Админ видит список всех `ChatSession` (даже удаленных пользователем).
*   **Просмотр Генераций:** Админ видит все записи о генерациях (`GenerationRecord`), включая метаданные.
*   **Просмотр Референсов:** Админ видит все загруженные изображения, используемые для редактирования (связь с `GenerationRecord`).
*   **Удаление Материалов:** Администратор может принудительно удалить записи из БД, даже если пользователь их "удалял" через UI.

### 4.2. Настройка Правил Автоначисления Баланса
Администратор должен иметь возможность настроить правила:
*   **Периодичность:** Ежедневно / Еженедельно / Ежемесячно.
*   **Сумма начисления:** Произвольная сумма (не ограничена фиксированным значением).
*   **Привязка к Группе:** Правило применяется ко всем пользователям, входящим в указанную группу AD.

### 4.3. Настройка Стоимости Услуг
Администратор должен иметь возможность изменять глобальные коэффициенты (см. п. 3.2 - 3.4).

## 🧠 6. Правила AI Agent (Для разработчика)

При работе над проектом, ваш AI-агент должен строго следовать этим директивам:

1.  **Транзакционная Целостность:** Все операции с балансом и записями должны быть атомарными в PostgreSQL.
2.  **Изоляция Процессов (Modularity):** Логика должна быть разделена на сервисы/классы (например, `AuthService`, `EconomyService`, `ChatService`, `ComfyUIService`). FastAPI должен выступать как **Controller**, вызывающий эти изолированные сервисы.
3.  **Паттерны Проектирования:** Использовать паттерн **Repository** для абстракции доступа к БД и **Strategy Pattern** для выбора метода расчета стоимости (LLM vs Image Gen vs Video Gen).
4.  **Модульность Расширения:** Все интеграции должны быть реализованы через интерфейсы/абстрактные классы, чтобы в будущем легко заменить:
    *   `LMStudioAdapter` $\rightarrow$ `TelegramBotAdapter`.
    *   `PostgresRepository` $\rightarrow$ `MongoRepository`.
5.  **Логика Генерации:**
    *   **Z-Image (Generator):** Принимает только `Prompt` + (опционально) `Base_Workflow`. Выдает одно изображение.
    *   **Qwen Image Edit (Editor):** Принимает `Prompt` + **(1, 2 или 3)** `Reference_Images`. Выдает отредактированное изображение.
